# Quick Boot 与 ruoyi-vue-pro 的差异说明

Quick Boot 基于 [YunaiV/ruoyi-vue-pro](https://github.com/YunaiV/ruoyi-vue-pro) 进行精简、整理和二次封装。本文用于说明当前 Quick Boot 仓库的定位、模块取舍和适用场景，帮助开发者判断是否适合作为自己的后端基础工程。

## 项目定位

`ruoyi-vue-pro` 是一个功能完整的企业级快速开发平台，覆盖权限认证、系统管理、基础设施、工作流、商城、CRM、ERP、MES、AI、IoT 等大量业务能力，更适合作为一套“大而全”的业务平台或企业应用底座。

Quick Boot 的定位更窄：它优先保留 Java 17 + Spring Boot 3 后端脚手架所需的基础能力，聚焦后端工程结构、starter 组件、系统管理、基础设施模块和本地开发体验。它不是上游项目的完整替代品，也不追求一次性覆盖所有业务套件。

## 核心差异

| 维度 | ruoyi-vue-pro | Quick Boot |
| --- | --- | --- |
| 项目目标 | 企业级全功能快速开发平台 | 轻量后端基础脚手架 |
| 功能范围 | 覆盖多业务域和大量应用场景 | 保留后端通用基础能力和核心模块 |
| 模块复杂度 | 模块多，业务覆盖面广 | 模块边界更集中，便于按需扩展 |
| 启动门槛 | 更适合直接使用完整平台能力 | 更适合先搭建后端基础，再按项目需要补充业务 |
| 二次开发重点 | 在完整平台上增加业务功能 | 在清晰后端底座上组合 starter 和模块 |
| 开源维护侧重 | 平台能力持续演进 | 文档、测试、CI、安全检查和本地开发体验逐步完善 |

## Quick Boot 当前保留的重点

Quick Boot 当前主要保留和整理以下能力：

- **后端多模块工程结构**：通过 `quick-dependencies`、`quick-framework`、`quick-module-system`、`quick-module-infra` 和 `quick-server` 拆分依赖管理、框架组件、业务模块和服务入口。
- **基础 starter 组件**：保留 Web、Security、MyBatis、Redis、WebSocket、Monitor、Job、MQ、Excel、Protection、多租户、数据权限、IP 等常用后端能力。
- **系统与基础设施模块**：保留用户、权限、租户、配置、文件、日志、代码生成、任务调度、监控等后端常用模块。
- **Spring Boot 3 技术栈**：统一使用 Java 17、Spring Boot 3、Maven、多模块依赖管理和环境变量占位配置。
- **公开仓库维护能力**：补充 README、CONTRIBUTING、SECURITY、CHANGELOG、CI、安全扫描和本地配置模板，降低开源维护和贡献门槛。

## Quick Boot 已裁剪或弱化的方向

Quick Boot 会有意识地减少以下复杂度：

- **大型业务套件**：暂不把工作流、商城、CRM、ERP、MES、AI、IoT 等上游完整业务域作为默认主线维护目标。
- **前端全量工程**：当前仓库聚焦后端脚手架，不默认包含完整前端管理端。
- **强平台化配置**：优先保持后端基础工程可理解、可裁剪、可复用，避免一开始引入过多业务耦合。
- **一次性全量覆盖**：文档、测试、Docker Compose、starter 示例和 release 自动化会按维护计划逐步补齐，而不是在初始阶段堆积过大范围改动。

## 更适合使用 Quick Boot 的场景

如果你的目标符合以下情况，Quick Boot 更适合作为起点：

- 需要快速启动一个 Java 17 + Spring Boot 3 后端项目。
- 希望复用权限、Redis、MyBatis、任务调度、监控、WebSocket、多租户或数据权限等基础能力。
- 项目业务边界尚未完全确定，需要一个可裁剪、可扩展的后端底座。
- 团队希望基于较小的仓库逐步补充测试、文档、CI 和安全检查。
- 你更关注后端 API 服务、SaaS 管理后台、内部管理系统或基础设施组件复用。

## 更适合直接使用 ruoyi-vue-pro 的场景

如果你的目标符合以下情况，直接使用上游项目通常更合适：

- 需要完整前后端一体的企业级快速开发平台。
- 希望开箱即用工作流、商城、CRM、ERP、MES、AI、IoT 等完整业务模块。
- 团队能够接受更大的模块规模，并希望基于成熟平台能力直接扩展业务。
- 项目需求和上游现有业务套件高度匹配，不需要明显裁剪。

## 维护原则

Quick Boot 后续会按以下原则继续演进：

- 保留对上游项目的致谢和协议说明。
- 优先补齐后端脚手架的文档、测试、CI、安全检查和本地启动体验。
- 每次改动保持聚焦，避免无关业务模块一次性大规模迁入。
- 新增能力优先以 starter、模块文档或最小可运行示例的方式沉淀。
- 对与上游差异较大的设计，优先在文档中说明取舍原因和适用边界。
